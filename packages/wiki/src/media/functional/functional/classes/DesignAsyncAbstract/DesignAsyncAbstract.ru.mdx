# Класс DesignAsyncAbstract

Асинхронный абстрактный базовый класс для управления свойствами конструктора и изменениями состояния. Расширяет `DesignAbstract` поддержкой асинхронной инициализации событий, что делает его идеальным для случаев, требующих загрузки данных или асинхронных вычислений.

## Ключевые особенности

- **Асинхронная инициализация** — поддержка инициализации событий на основе `Promise` через метод `initEvent`.
- **Управление свойствами** — наследует структурированную обработку входных данных (`props`) и хранилища событий (`event`).
- **Асинхронный рабочий процесс** — гарантирует, что обратные вызовы запускаются только после завершения асинхронной подготовки состояния.

## Инициализация

```typescript
import { DesignAsyncAbstract } from '@dxtmisha/functional'

interface MyProps { id: string }
interface MyEvent { name: string }

class MyAsyncDesign extends DesignAsyncAbstract<MyProps, MyEvent> {
  protected async initEvent(): Promise<void> {
    const data = await fetch(`/api/user/${this.props.id}`).then(r => r.json())
    this.event.name = data.name
  }
}

const design = new MyAsyncDesign({ id: '123' }, (event) => {
  console.log('Асинхронные данные обновлены:', event)
})
```

## Управление данными

### `make`

Инициирует асинхронный процесс обновления. Вызывает `makeCallback` (не дожидаясь завершения) и возвращает экземпляр для цепочки вызовов.

**Параметры:**
- `compelled?: boolean` — если `true`, принудительно обновляет данные независимо от наличия изменений.

**Возвращает:** `this`

### `makeCallback`

Основная асинхронная логика выполнения для обновления состояния. Дожидается завершения `initEvent` перед запуском обратных вызовов и обновлением трекера изменений.

**Параметры:**
- `compelled?: boolean` — принудительно обновляет данные.

**Возвращает:** `Promise<void>`

## Внутреннее устройство

`DesignAsyncAbstract` использует тот же механизм `DesignChanged`, что и его синхронный аналог, но оборачивает жизненный цикл в `async/await`. Это гарантирует, что даже при сложной асинхронной логике настройки состояние остается согласованным, а изменения точно отслеживаются перед уведомлением слушателей.
