# AI Package Memory: @dxtmisha/constructor

## Component Creation & Styling Rules
1. **Types generation**: Never manually put properties inside the `// :classes [!] System label / Системная метка` block in `types.ts`. Furthermore, never delete these system labels. Anything inside these system labels is generated by scripts.
2. **Properties mapping**: The properties inside the system labels in `types.ts` are automatically generated based on `properties.json`. To add a new class (e.g. `square: string`), create or edit `properties.json` inside the `src/constructors/<Component>` folder with `{"#square": {}}`.
3. **Execution**: After updating `properties.json`, run `npm run constructor` inside the `packages/constructor` directory to let the script automatically populate `types.ts` and other auto-generated types.

## Code Style & Refactoring Rules (Developer Feedback)
4. **No Abbreviations**: Strictly adhere to the project's zero-tolerance policy on abbreviations. Never use variable names like `el` (rename to `squareElement` or `element`), `idx` (rename to `index`), `temp`, `val`, `dx`, etc.
5. **Layout Coordinates**: When updating coordinate values or layout points, prefer the `ImageCoordinator` object type (`{ x: number, y: number }`) over separate `(x: number, y: number)` arguments.
6. **Dynamic Property Mapping**: Do not hardcode local CSS variables (such as `--_sa-*`). Dynamic position styles must be mapped using a `property` dictionary built from the component name (e.g., `--${className}-sys-square-x`) to fit the design token architecture.
7. **CSS Variable Retrieval**: Retrieve layout variables in SCSS templates using the `ui.v('??sys-*')` wrapper, aligning with standard system variable routing.
8. **Descriptive State Names**: Do not use vague or generic variable names like `old` or `save` for selection states. Use descriptive names like `selectedStart` (saved initial selected items when interaction begins) and `selectedCurrent` (currently accumulated selection elements).
9. **Alphabetical Sorting**: In class constructors that accept a `constructors` parameter for dependency injection, all related lists (TSDoc parameters, constructor parameter types, destructuring, and property assignments) must be sorted alphabetically.
10. **Component Inheritance**: Child components (like `MenuLanguage` or `Chip`) must directly inherit their core logic classes (`class Child extends Parent`), design constructors (`class ChildDesign extends ParentDesign`), properties (`ChildPropsBasic` extending/aliasing `ParentPropsBasic`), and type definitions (`ChildComponents`, `ChildEmits`, `ChildSlots`, `ChildExpose` inheriting/mapping from their parent equivalents) rather than duplicating them. They must also define their own `<Name>Include` classes and `<Name>PropsInclude`/`<Name>ComponentInclude` types following the parent's structure.
